# 步驟一：決定題目並撰寫專案基本資訊

## 1. 專案題目
- 專案名稱：逢甲吃吃喝喝平台

## 2. 背景描述

背景描述：每次下課飢腸轆轆之時，經常會找不到有什麼好吃的，想解決窮大學生和初來乍到的旅人不知在逢甲能吃什麼的煩惱，且讓大家都能快速找到便宜又美味的美食。
## 3. 預期使用者

預期使用者：
* 逢甲周邊飢腸轆轆的居民
* 逢甲大學的學生
* 不熟悉逢甲的外地人


## 4. 想解決的問題

想解決的問題：生存

## 5. 初步需求

初步需求：解決不知道吃什麼又怕被割韭菜的生活

# 步驟二：使用 Prompt 請 AI 擴充需求

## 1. 提供給 AI 的內容

請將上方「步驟一」整理好的內容貼給 AI。

## 2. 使用的 Prompt

請貼上你們實際詢問 AI 的 Prompt。

```text
以下是一個專案的基本資訊與初步需求，請協助我分析這個專案還可能需要哪些需求。

請根據專案背景、預期使用者與使用情境，
補充這個系統可能還需要的需求，並說明原因。
請先不要省略任何你認為重要的需求。
```

## 3. AI 回覆內容

## AI 回覆：
1. 核心篩選需求：精準抗通膨對於學生和居民來說，「預算」與「飽足感」是最高優先級。
* CP 值（性價比）量化指標：
原因：一般的 5 星評分太籠統。系統應增加「吃飽程度」與「價格比」的權重，例如標註「100 元內可吃飽」或「加飯不用錢」。
* 動態營業狀態即時顯示：
原因：大學生常在深夜或離峰時段覓食，最崩潰的是到了現場發現沒開。需整合 Google Maps API 或讓店家/用戶回報即時營業狀態。
* 精準預算分類：
原因：將分類細化為「銅板美食（<$60）」、「學生正餐（$60-$120）」、「聚餐首選（>$200）」。
2. 社群與信任需求：反制「韭菜」陷阱要避免被割韭菜，必須建立在「真實性」之上。
* 學生認證/真實評論機制：
原因：外地旅人常被業配文誤導。系統可設計「逢甲人推薦」標籤（需學生 Email 認證），增加可信度。
* 黑名單與「排雷」功能：
原因：直接列出「漲價過兇」或「衛生堪憂」的負面回饋，讓使用者能快速避開雷區。
* 即時優惠/剩餘食份資訊：
原因：結合剩食惜食（如：晚間 8 點後的折扣），既解決生存問題又環保，店家也能清庫存。
3. 功能性與場景需求：提升便利性針對不同使用者的情境（如趕課、獨自用餐、朋友聚會）。
* 「今天吃什麼」隨機轉盤：
原因：針對有「選擇困難症」的使用者，根據設定的預算隨機推薦一家評價優良的店。
* 步行距離與擁擠程度：
原因：學生下課只有 10-20 分鐘，需顯示從「商學大樓」或「資電館」走過去的預估時間。
* 單人友善標籤：
原因：許多學生是獨自進食，需標註是否有單人座位或是否排斥單人用餐。
4. 營運與獲利潛力（隱藏需求）為了讓平台能長久生存，建議考慮：
* 店家認領與優惠發布：
原因：讓優質小店能主動發布「憑學生證折 5 元」等活動，增加平台流量。
* 離峰時段導流：
原因：解決熱門店排隊過久的問題，推薦目前不用排隊的替代方案。

請簡要回答以下問題：

1. 在這次活動中，AI 對你們最大的幫助是什麼？
在這次活動中，AI最大的幫助是幫助我們快速發想更多功能與需求。一開始我們只想到像是搜尋店家或評價餐廳這種比較基本的功能，但透過AI的建議，我們多想到像是今天吃什麼隨機轉盤、剩餘座位回報、CP值篩選、黑名單提醒等功能。這些想法讓我們可以從不同角度去思考使用者在逢甲找食物時可能遇到的問題，也讓整個平台的設計變得更完整。
2. 你們覺得 AI 提出的內容有哪些地方需要人來判斷？
雖然AI可以提供很多建議，但有些功能不一定符合實際情況，所以還是需要人去判斷。例如AI提出物價追蹤即時通報和共乘分攤功能，但我們討論後覺得逢甲附近的店家距離其實都不遠，共乘的需求應該不高，而餐點價格也不會常常改變，所以即時追蹤的必要性不大。因此最後我們選擇刪除或修改這些需求。這也讓我們知道AI提供的是參考，最後還是需要人去思考實際使用情境。
3. 你們從這次活動中學到什麼？
透過這次活動，我們學到AI可以幫助我們快速整理想法，但不能完全依賴它。在設計系統需求時，還是需要透過小組討論，從使用者角度去思考哪些功能真的有幫助。例如我們覺得這個平台通常會在肚子餓、趕時間、不知道吃什麼的情況下使用，所以最重要的是讓系統簡單、好操作、能快速找到店家，而不是一直增加很多複雜的功能。
